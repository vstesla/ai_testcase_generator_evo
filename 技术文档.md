# 智能测试用例与附件生成系统 (AI TestCase Generator EVO) - 项目经验复盘

## 📝 项目背景与概述
**项目描述**：本项目是一个专为金融托管与清算业务打造的智能化测试辅助平台。系统基于前后端分离架构，对接底层 AI Agent 服务（DeepSeek-v4-flash），提供测试数据的自动化泛化与对抗攻击样本生成；支持各类复杂指令附件（如缴款通知书、标的合同）的动态排版与批量生成；并深度融合 OCR 引擎，实现生成附件与预期结构的自动化比对评测。
**业务价值**：将传统人工构造复杂金融测试用例的时间成本降低 90% 以上，解决了大批量测试附件手动排版效率低下的痛点，并通过闭环的自动化评测机制显著提升了事中规则校验的覆盖率。

## 🛠 核心技术栈
- **后端**：FastAPI, Python 3.12, Celery, Redis, PyMySQL, ReportLab, Playwright, Tenacity
- **前端**：React, Ant Design, Axios, XLSX.js
- **基础设施**：Docker, 腾讯云 COS 对象存储, DeepSeek API (JSON Output)

## ✨ 项目核心亮点与贡献

1. **全栈架构设计**：基于 React + FastAPI 独立构建前后端分离系统。前端利用 Ant Design 实现智能表单联动与历史数据多维检索；后端集成 Celery + Redis 异步队列，支持 300+ 份金融单据的并行生成与状态轮询。
2. **智能化物料管道**：开发基于 ReportLab 的 PDF 动态渲染引擎，支持缴款通知书、标的合同等多种非标指令的高精度排版，并利用 DeepSeek/内部 AI Agent 实现对抗性测试样本的自动化注入。
3. **AI 自动化评测闭环**：打通生成产物与 OCR 解析引擎接口，设计并实现一套针对字段级的 NLP 比对算法（包含数值规范化与模糊匹配），自动产出准确率报告，实现了“生成即评测”的质量闭环。
4. **工程化性能攻关**：利用 `asyncio` 锁机制解决高并发下的主键冲突；采用 `BytesIO` 内存流取代传统落盘模式，降低 I/O 损耗并规避了生成海量 PDF 时的内存溢出（OOM）风险。
5. **稳定性保障**：集成 `Tenacity` 指数退避策略优化 AI 网关调用链路，并在前端实现多文件自动合并下载/ZIP 自动打包逻辑，显著提升用户体验。

## 🚀 攻坚的技术难点与解决方案

### 模块一：后端高并发轮询下的 504 宕机与连接池泄漏
- **问题描述**：在前端发起多任务轮询（查询任务状态）时，后端频繁报出 `504 Gateway Timeout`，甚至导致整个 API 服务冻结，无法响应任何新请求。
- **发现过程**：
  1. 通过排查网络面板，发现瞬间存在数十个 `generation_result` 并发请求卡在 Pending 状态。
  2. 深入审查 FastAPI 路由层代码，发现轮询接口错误地使用了 `async def` 声明，但其内部调用的 `pymysql` 数据库查询却是纯同步阻塞 I/O。这导致单线程的事件循环（Event Loop）被完全卡死。
  3. 进一步排查数据库连接模块 `mysql_connector.py`，发现在 `execute_query` 的执行逻辑中，仅在异常捕获 `except` 块中归还了连接，而在正常执行完毕后直接 `return`，导致连接池（大小为 20）被瞬间抽干。
- **解决思路**：
  1. **释放事件循环**：将包含同步数据库 I/O 的 FastAPI 路由从 `async def` 修改为 `def`。利用 FastAPI 底层机制，将同步阻塞任务自动卸载（Offload）到外部线程池中并发执行，彻底解放主事件循环。
  2. **连接池闭环管理**：重构 MySQL 封装类，强制引入 `finally` 代码块，确保无论 SQL 执行成功或抛出异常，均严格调用 `self.pool.release_connection(conn)` 释放连接，实现了连接池的稳定流转，彻底消除了 504 宕机现象。

### 模块二：MySQL 事务隔离级别导致的“状态幽灵读”
- **问题描述**：后台 Celery 任务已执行完毕并将数据库状态更新为成功（`Y`），甚至手动在数据库修改状态，但前端轮询接口始终返回“处理中（`P`）”，状态产生严重滞后。诡异的是，只有在新触发一次生成任务后，旧任务的状态才会突然刷新。
- **发现过程**：
  1. 首先排查前端是否存在数据缓存，确认前端确实获取到了后端的最新响应，证明问题在后端或数据库层。
  2. 分析触发刷新的诡异现象：新任务的 `INSERT/UPDATE` 操作会附带执行 `commit()`，执行后旧任务数据随之刷新。
  3. 锁定根因为**数据库事务隔离级别（Transaction Isolation）**：`pymysql` 连接配置了 `autocommit=False`，且 MySQL 默认为“可重复读（Repeatable Read）”。FastAPI 在第一次 `SELECT` 轮询时开启了事务快照，由于后续只有 `SELECT` 没有 `commit`，导致接口永远在读取旧的事务快照（幽灵读）。
- **解决思路**：
  在 `mysql_connector.py` 的连接池初始化中，将 `autocommit` 显式设置为 `True`。确保每次轮询的 `SELECT` 查询都能获取到数据库最新已提交的数据，保证了前后端状态的毫秒级同步。

### 模块三：React 闭包陷阱与定时器雪崩效应
- **问题描述**：前端出现极其严重的并发风暴。页面初次加载或刷新时，Network 面板瞬间爆出 90+ 个相同的轮询请求；同时，页面列表状态互相覆盖，新生成的任务状态会把上一条任务的状态“吃掉”。
- **发现过程**：
  1. 审查前端请求链路，发现在页面初始化拉取历史记录（默认 100 条）时，由于缺乏活跃时间限制，代码直接为所有未完成比对的历史遗留任务盲目启动了轮询定时器，导致初始化瞬间引发并发雪崩。
  2. 分析定时器内部逻辑，发现判断成功后未能通过 `return` 阻断执行流，导致定时器无限嵌套递归。
  3. 排查状态覆盖问题，发现轮询回调函数在 `setTimeout` 内部捕获了初次渲染时的 State 引用（典型的 React Stale Closure 闭包陷阱），导致每次更新都基于旧数组进行覆盖。
- **解决思路**：
  1. **防抖与活跃期拦截**：在初始化遍历逻辑中引入时间窗口校验，仅对“最近 30 分钟内活跃”的记录恢复轮询，屏蔽历史脏数据的干扰。
  2. **彻底清理幽灵定时器**：重构 `clearTimeout` 逻辑，修复删除对象引用时的条件作用域错误；并在任务到达终态（Y/N）时强制 `return` 中断递归。
  3. **纯函数状态更新**：将所有的 `setGeneratedResults(newState)` 重构为函数式更新 `setGeneratedResults(prev => prev.map(...))`，确保每次状态修改都基于 React 最新的状态队列，彻底解决了状态相互覆盖的渲染 Bug。

### 模块四：Celery 多进程模型与异步事件循环的系统级冲突
- **问题描述**：引入 Celery 处理耗时文档生成任务后，在 Windows 本地开发环境下频繁抛出 `RuntimeError: There is no current event loop in thread` 导致协程崩溃；同时，后台进程执行 COS 文件上传时报 `AccessDenied`（环境变量丢失）。
- **发现过程**：
  1. 分析报错堆栈，确认报错源于 Celery 默认的 Eventlet/Prefork 并发池与 Python 原生的 `asyncio` 在非 Unix 系统下产生了底层的线程机制冲突。
  2. 打印 Worker 进程的环境变量，发现 Celery 在 Windows 下孵化的子进程并未继承主进程加载的 `.env` 配置，导致鉴权失败。
- **解决思路**：
  1. **协程生命周期隔离**：在 `tasks.py` 中为异步任务手动接管事件循环，使用 `loop = asyncio.new_event_loop()` 显式创建、设置并在 `finally` 中安全销毁独立循环，隔离线程干扰。
  2. **并发池降级**：结合本地物理机开发环境特性，将 Celery 启动参数调整为 `-P threads`，规避了 Windows 下的进程分叉问题。
  3. **全局环境变量注入**：在 `celery_app.py` 顶层强制执行 `load_dotenv()`，确保所有的 Worker 在启动初始化时能够正确装载云服务鉴权密钥。

### 模块五：大模型脏数据注入与解析引擎崩溃防御
- **问题描述**：在使用大模型进行对抗攻击样本生成时，底层 `openpyxl` 写入 Excel 时抛出 `IllegalCharacterError` 崩溃。
- **发现过程**：
  排查模型生成的 JSON 日志，发现在“对抗攻击”模式下，大模型不仅生成了预期的 XSS 脚本（如 `<script>alert('xss')</script>`），还夹带了肉眼不可见的 XML/ASCII 控制字符（如 `\013`, `\014` 等），超出了 Excel 标准格式的支持范围。
- **解决思路**：
  1. **清洗拦截层**：在将大模型生成的数据写入 DataFrame 之前，引入正则表达式 `re.compile(r'[\000-\010]|[\013-\014]|[\016-\037]')` 作为清洗拦截层，精准剔除所有非法的不可见控制字符，保留有效的对抗攻击测试载荷。
  2. **输出规范化**：升级底座模型调用参数，通过设置 `response_format={'type': 'json_object'}` 强制开启大模型的结构化输出模式，大幅降低了解析异常的概率，增强了数据管道的鲁棒性。
